Git & GitHub Command Reference

Learn Git commands
and when to apply them

A practical reference covering the commands you'll reach for most, with context on why and when each one matters, not just what it does.

01
Setup & Configuration
First things you run on a new machine or repository
git config --global user.name "Your Name"
Set global username
Identifies your commits across all repos on this machine.
once
git config --global user.email "you@example.com"
Set global email
Must match your GitHub email for commits to link to your profile.
once
git init
Initialise a repository
Creates a hidden .git folder. Run once inside a new project folder.
once per project
git clone <url>
Clone a remote repository
Downloads the full repo and sets up origin automatically.
remote

When to use this section

  • New machine: run the config commands before anything else
  • Starting a project from scratch: git init inside your project folder
  • Joining an existing project: git clone instead of init
02
The Basics — Stage & Commit
The core daily loop: track changes and save snapshots
git status
Check working tree status
Shows modified, staged, and untracked files. Run this constantly — it's free.
read-only
git add <file>
Stage a specific file
Moves changes to the staging area. Only staged changes are included in the next commit.
git add .
Stage all changes
Stages every modified and new file. Convenient but review with git status first.
git commit -m "message"
Commit staged changes
Creates a permanent snapshot. Write the message in present-tense imperative: "Fix login bug".
git log --oneline
View commit history (compact)
Shows abbreviated hash + message. Add --graph to see branch visualisation.
read-only
git diff
Show unstaged changes
Line-by-line diff of what changed but isn't staged yet. Use git diff --staged to see staged changes.
read-only

When to use this section

  • Before starting work: git status to understand current state
  • Finished a logical unit of work: add + commit to save a snapshot
  • Reviewing your work: git diff before staging, git log to review history
03
Branching & Merging
Isolate work, switch context, and bring changes together
git branch
List branches
Shows all local branches. Current branch is marked with *.
read-only
git branch <name>
Create a branch
Creates the branch but doesn't switch to it. Use checkout -b to do both at once.
git checkout -b <name>
Create & switch to branch
The most common way to start new work. Equivalent to git switch -c in modern Git.
git checkout <branch>
Switch branch
Updates working directory to match that branch. Commit or stash first if you have changes.
git merge <branch>
Merge a branch
Brings changes from <branch> into your current branch. May create a merge commit.
git branch -d <name>
Delete a branch
Safe delete — Git warns you if the branch hasn't been merged. Use -D to force.
destructive

When to use this section

  • Starting any new feature or fix: always branch off main first
  • Context-switching: stash or commit, then checkout to another branch
  • After a PR is merged: delete the feature branch locally to keep things clean
04
Working with Remotes
Push, pull, and stay in sync with GitHub
git remote -v
List remotes
Shows the fetch and push URLs. origin is the default name for the GitHub remote.
read-onlyremote
git push origin <branch>
Push a branch to GitHub
Uploads your local commits. First push of a new branch often needs -u origin <branch> to set upstream.
remote
git pull
Pull latest changes
Fetches and merges remote changes into your current branch. Equivalent to fetch + merge.
remote
git fetch
Fetch without merging
Downloads remote changes but doesn't apply them. Safer than pull — lets you inspect first.
read-onlyremote
git push -u origin <branch>
Push & set upstream (first push of a new branch)
The -u flag links your local branch to the remote one. After this, plain git push and git pull will work without specifying the branch name.
remote

When to use this section

  • Start of a work session: git pull to get the latest from teammates
  • Finished a feature: git push then open a Pull Request on GitHub
  • Before merging: git fetch and review what changed upstream
05
Undoing Changes
Fix mistakes — ranging from safe to permanent
git restore <file>
Discard working directory changes
Reverts a file to its last committed state. Unstaged changes are lost permanently.
destructive
git restore --staged <file>
Unstage a file
Removes a file from the staging area without touching the actual file content.
safe
git commit --amend
Amend the last commit
Rewrite the most recent commit message or add forgotten files. Only use before pushing.
rewrites history
git revert <hash>
Revert a commit (safe)
Creates a new commit that undoes a specific commit. Safe to use on shared branches.
safe on shared
git reset --soft HEAD~1
Undo last commit, keep changes staged
Moves HEAD back one commit. Your files and staged changes are preserved.
rewrites history
git stash
Stash uncommitted changes
Temporarily shelves changes so you can switch context. Restore with git stash pop.
reversible

When to use this section

  • Typo in last commit message: --amend (before pushing only)
  • Need to undo a pushed commit: always use git revert
  • Need to switch branches mid-work: git stash, switch, then stash pop
  • Staged the wrong file: git restore --staged
06
Advanced Commands
Rebase, cherry-pick, and powerful history tools
git rebase <branch>
Rebase onto a branch
Replays your commits on top of another branch. Creates a cleaner, linear history than merging.
rewrites history
git rebase -i HEAD~N
Interactive rebase (last N commits)
Opens an editor to squash, reorder, reword, or drop the last N commits before sharing.
rewrites history
git cherry-pick <hash>
Cherry-pick a commit
Applies a single commit from another branch to your current branch. Useful for hotfixes.
git bisect start
Binary search for a bug
Helps you find which commit introduced a bug using binary search. Mark commits as good or bad.
read-only
git tag v1.0.0
Create a tag
Marks a commit with a label, typically for releases. Push tags with git push --tags.
git log --all --decorate --oneline --graph
Visualise full branch graph
Nicknamed git adog. Shows a full ASCII graph of all branches and commits.
read-only
07
Update a Repository
If you already have a repository cloned locally and have made commits in the past, follow these standard steps to add new commits and push them to GitHub.
Feature Branch Workflow
git status
→
git add .
→
git commit -m "Update navigation bar styling"
→
git pull origin main
→
git push origin main
git status
1. Check what has changed.
Shows modified, staged, and untracked files. Always run this first to see what needs to be committed.
git add .
2. Stage all changes
Adds all new, modified, and deleted files in the current directory to the staging area. Prepares everything for commit.
git add <file>
Stage a specific file
Stages only the specified file(s). Replace <file> with index.html or src/app.js. Repeat for multiple files.
3. git commit -m <message>
Commit staged changes
Creates a new commit with all staged changes. Use present tense, e.g., "Add search feature" or "Fix typo in footer".
git commit -am <message>
Stage tracked files and commit
Automatically stages all already tracked files and commits them in one command. Does not include new/untracked files.
git pull origin <branch>
4. Pull latest changes from remote
Downloads and merges the latest commits from GitHub into your current branch. Run before pushing to avoid conflicts.
git pull --rebase origin <branch>
Pull with rebase (cleaner history)
Downloads remote changes and replays your local commits on top. Creates a linear history instead of merge commits.
git push origin <branch>
5. Push commits to GitHub
Uploads your local commits to the remote repository. Replace with main, master, or your feature branch name.
git push
Push to upstream branch
If upstream tracking is set (using -u on first push), this pushes to the correct remote branch automatically.
git log --oneline -5
View recent commits
Shows the last 5 commits with shortened hashes and messages. Useful to confirm your commit is ready to push.
git diff
See unstaged changes
Shows line-by-line differences between your working directory and the last commit. Review before staging.
git diff --staged
See staged changes
Shows what will be included in your next commit. Review before running git commit.
08
Typical GitHub Workflow
The everyday feature branch → Pull Request loop
Feature Branch Workflow
git pull
→
checkout -b feature
→
add + commit
→
push -u origin
→
Open PR on GitHub
→
merge & delete branch
git pull origin main
1. Sync with main
Always start from an up-to-date main branch to avoid large merge conflicts later.
git checkout -b feature/my-feature
2. Create a feature branch
Keep branches small and focused. A good branch name describes the change: fix/login-redirect.
git add . && git commit -m "feat: add X"
3. Commit your work
Commit often and in small logical chunks. Use conventional commits: feat:, fix:, docs:.
git push -u origin feature/my-feature
4. Push your branch
GitHub will show a prompt to open a Pull Request as soon as you push.
remote
git checkout main && git pull
5. After PR is merged
Switch back to main and pull the merged changes so your local is up to date.
git branch -d feature/my-feature
6. Clean up the branch
Delete the local branch to keep your workspace clean. The remote branch can be deleted on GitHub.
destructive
09
First-Time Upload (New Repository)
Upload a local project to a new GitHub repository using HTTPS.
Feature Branch Workflow
git init
→
git add .
→
git commit -m "Initial commit"
→
git remote add origin https://github.com/your-username/your-repo-name.git
→
git branch -M main
→
git push -u origin main
git init
1. Initialize local Git repository.
Creates a .git folder in your project directory, enabling Git version tracking on your local machine. Run this inside your project folder.
git add .
2. Stage all changes
Adds all new, modified, and deleted files in the current directory to the staging area. Prepares everything for commit.
git commit -m "Initial commit"
3. Create first commit
Takes a snapshot of all staged files. The message "Initial commit" is standard for a project's first commit.
git remote add origin <https-url>
4. Link to GitHub repository
Connects your local repo to a remote GitHub repo via HTTPS. Replace with your GitHub repo URL (e.g., https://github.com/username/repo.git).
git branch -M main
5. Rename default branch to main
Renames the default branch from master to main. Matches GitHub's current default branch naming convention.
git push -u origin main
6. Push to GitHub and set upstream
Uploads your commit to GitHub and sets the upstream tracking. After this, future pushes can use just git push.
